## 4. Problem Statement

Currently, JTL Cloud use cases represent isolated workflows rather than a continuous, productively usable end-to-end journey. New customers (micro-retailers) cannot complete the entire process from sign-up to physical shipping without using the JTL-Wawi desktop client or the Shop backend for configuration steps.

**Risks of non-implementation:**

* **Competitive Disadvantage:** Customers migrate to providers with "one-click" onboarding (Shopify, Billbee).
* **No SaaS value proposition:** Without a seamless end-to-end path, the Cloud ERP cannot be validated as a real SaaS offering.
* **Complexity trap:** Development of niche features before the base "Buy-to-Ship" flow is stable.

## 5. Business Requirements

### 5.1 Functional Requirements

* **Central Onboarding Wizard:** Guided setup of company master data, infrastructure provisioning (Hosting, Shop), shipping provider, and payment method — all without leaving the ERP Cloud UI.
* **Master Data Pre-Fill (ISI Gateway):** Company data (general, bank, legal/tax) is retrieved from JTL-Kundencenter via ISI Gateway using the Customer ID (KID) and pre-filled into the wizard form.
* **Automated Infrastructure Provisioning:** JTL-Wawi instance and JTL-Shop hosting are spun up automatically. Shop URL and license key are registered in JTL-Wawi via API. Payment method "Purchase on Account" and shipping method "DHL Paket Inland" are created automatically.
* **Silent App Installation:** JTL-Shipping App is installed via Hub API (User JWT) and mapped to the DHL Paket Inland shipping method in JTL-Wawi — no manual interaction required.
* **Simplified Item Management:** Reduced input mask (Name, Price, Image, Description). Items can be activated for JTL-Shop sales channel directly from ERP Cloud. "Standard" item category is pre-selected by default.
* **Save & Sync Trigger:** Clicking "Save" persists the item first; only after successful persistence does the frontend trigger the shop sync endpoint (delta logic). The sync checks JTL Worker status — if active, manual sync is suppressed.
* **Order List with Automated Sync:** New shop orders appear in ERP Cloud automatically via Eventbus interval trigger (every 5 minutes). Status columns show Payment Status (Paid/Unpaid) and Shipping Status (Shipped/Not Shipped).
* **Invoice Creation & PDF Output:** Invoice is created in JTL-Wawi backend; PDF is transmitted via Gateway to the ERP frontend and opened in the browser print dialog.
* **Mark as Paid:** Full payment can be set or removed via guided interaction in Order Details.
* **Shipping Label Generation:** Sequential API chain: (1) Create delivery note via JTL-Wawi CloudAPI → (2) Request shipping label PDF directly from JTL Shipping App. Label is sent to browser print dialog. Only possible when order is paid OR payment method is "Purchase on Account" AND order contains only non-stock-managed items. (Architecture updated 2026-05-22: label generation goes directly from ERP Cloud to JTL Shipping App.)
* **Eventbus-based Shop Sync:** Interval sync (every 5 minutes) and real-time event-based sync (triggered on item save) — no JTL Worker.

### 5.2 Non-Functional Requirements

* **Time-to-Value:** Complete setup from sign-up to first item ready for sale must not exceed 120 minutes. Future target: 10 minutes (pre-spawning, not in MVP scope).
* **Platform Focus:** Desktop web browser only. No mobile support.
* **Zero-Exit Journey:** No login to JTL-Wawi desktop or JTL-Shop backend required for the core "Buy-to-Ship" path — with the exception of AGB / Terms & Conditions. In the wizard we link the merchant to the JTL-Shop backend for reviewing/updating AGB and explicitly state that JTL assumes no liability for unchecked default wording.
* **Feature Flag Protection:** The "Start with new instance" wizard path is only accessible when the feature flag is active for the user/tenant. The option cannot be bypassed via URL manipulation.

## 6. Suggested Solution Approach

Implementation of a radically simplified MVP flow for new micro-retailer customers. All technical complexity is abstracted away through background orchestration. The user interface is reduced to the minimum required to complete the "Buy-to-Ship" journey. All new features are gated by feature flags.

The ERP Cloud acts as the central orchestration layer — it initiates hosting provisioning, shop setup, shipping app installation, and sync triggers. Business logic remains in JTL-Wawi by architectural design; the ERP Cloud exclusively communicates via API.

## 7. User Stories

* **Story 1 – Onboarding:** As a micro-retailer (Sebastian Evers), I want to set up my entire system landscape (ERP Cloud + JTL-Shop + Shipping) automatically via a guided wizard, so that I can be ready to sell in under 2 hours without any technical ERP knowledge.
* **Story 2 – Item Creation:** As a micro-retailer, I want to create my first item through an intuitive guided flow, activate it directly for my JTL Shop, and start the sync automatically, so that I can get my first product live in the shortest possible time.
* **Story 3 – Order Processing:** As a micro-retailer, I want to process my shop orders, create invoices, and complete shipping including label printing through a guided interface, so that I can quickly deliver to my customers without prior ERP knowledge.

## 8. User Flow Design

### Phase 1 – Onboarding & Setup (Wizard)

1. **Login** via JTL-Hub → ERP App (erp.jtlcloud.com)
2. **(100) Pre-selection screen:** "Connect existing instance" or "Start with new instance" (feature-flag gated).
3. **(200) Company Data – General:** (ISI Gateway pre-fill: name, address, contact).
4. **(300) Company Data – Payment:** (ISI Gateway pre-fill: IBAN, BIC, tax data).
5. **(400) Shipping Checkbox:** User confirms shipping terms; single confirmation step.
6. **(500) Shop Checkbox + Shop Name:** User agrees to online shop terms; acknowledges liability notice re. AGB; shop name pre-filled with company name (editable).
7. **(600) Setup Progress Screen:** "SQL Server Principle" (all input collected upfront, single consolidated setup phase):
* Set up shipping
* Boot up shop
* Set up ERP


8. **(700) Success Screen:** "Your business is ready!" → CTA: "Create your first item now".

### Phase 2 – Product Setup

* Wizard CTA "Create your first item now" → **Item creation form directly** (no intermediate overview).
* Fill in Name, Price, Image, Description, activate Sales Channel.
* Save & Transfer → Redirect to item overview.
* Eventbus sync trigger → Item appears in JTL-Shop.

### Phase 3 – Shipping & 1:1 Delivery (Buy-to-Ship)

* Orders menu → Order list (interval sync).
* Order details (payment status, invoice creation).
* Mark as Paid → Ship Order → Generate Shipping Label → Browser print dialog.
* Order marked as "Shipped" → Eventbus sync updates shop status.

## 9. UI/UX Design

* **Core Principle:** No rollback of existing masks from Q1. The UI maintains its state; new wizard and optimizations focus exclusively on the "Buy-to-Ship" path.
* **Main Navigation:** Dashboard / Items / Orders / Invoices (legacy overview).
* **Shipping Action:** One-click process. "One Order = One Package" — no package quantity prompt in MVP.
* **Tracking:** Integration of a link to the branded JTL-Shipping tracking page in the order list.

## 11. Out of Scope

| Area | Details |
| --- | --- |
| **Inventory Management** | No stock tracking; stock-managed items cannot be shipped via Cloud ERP |
| **Item Complexity** | No variants, BOMs, configurable items, serial numbers, batch/BBD tracking |
| **Price Labeling** | No unit prices, customer group prices, or platform-specific pricing |
| **Payment Gateways** | No external providers (PayPal, etc.); "Purchase on Account" only |
| **GPSR Compliance** | No dedicated fields; info entered manually in description |
| **Packaging Regulation** | No weight or packaging material fields |
| **Mobile Support** | Desktop browser only |
| **Partial Shipments** | Orders fulfilled completely (1:1 only) |
| **Shipping Label Reprint** | Done via JTL-Shipping App; not in ERP Cloud UI |
| **Post-Wizard Config** | Company/Payment/Shipping methods cannot be changed after completion (TCR-1190 planned) |
| **Domain Management** | No automated purchase/DNS; shop provisioned under standard URL |
| **Standard T&Cs via API** | User informed via success screen to review T&Cs in JTL-Shop backend |
| **Imprint** | Updated manually by user in JTL-Shop backend |
| **Invoice Detail View** | Overview carried over from Q1; no detail view in this initiative |
| **Multi-Shop Support** | Additional sales channels beyond JTL-Shop not in scope |
